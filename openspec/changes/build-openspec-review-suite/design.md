## Context

目标是构建一个应用套件：CLI 负责声明“哪些本地目录被纳入 OpenSpec 阅读范围”，Web 负责对这些目录中的 OpenSpec 工件进行关系化预览。该套件的核心不是通用 Markdown 浏览，而是紧扣 OpenSpec 中 proposal、design、tasks、spec 的结构关系，支持 spec 阶段的推进式审阅。

## Goals / Non-Goals

**Goals:**
- 提供稳定的多工作目录绑定机制，支持增删查与解绑。
- 提供以 Change 为主轴的审阅工作台，优先展示需要及时审阅的内容。
- 提供目录双模式（文件树 / Spec 卡片）与阅读区多模式（串读 / 并排）。
- 在主流程中常驻 Capability 全局视角，避免局部推进时失去全局认知。
- 全程只读，不引入在线编辑复杂度。

**Non-Goals:**
- 不支持在 Web 端直接编辑 OpenSpec 文档。
- 不在首版引入复杂评分或机器学习优先级算法。
- 不覆盖 OpenSpec 以外的通用文档管理能力。

## Decisions

### Decision 1: 产品主轴采用 Change-first
- 方案：首页默认显示 Change 推进队列，以“是否需要及时审阅”驱动排序与进入阅读。
- 原因：用户目标是 spec 阶段快速推进，优先回答“现在先审哪一个”。
- 备选：Capability-first 首页。被拒绝原因是推进节奏弱、难形成执行路径。

### Decision 2: Capability 以常驻辅助面板出现
- 方案：阅读区右侧常驻 Capability Lens，显示当前 Change 影响哪些 Capability，并提供跳转。
- 原因：推进过程中保持全局视角，但不抢主流程焦点。
- 备选：单独页面查看 Capability 全局。被拒绝原因是上下文切换成本高。

### Decision 3: 双浏览模式 + 双阅读模式
- 方案：目录区支持文件浏览模式与 Spec 卡片模式；阅读区支持 Change Pack 串读与并排对照。
- 原因：既服务“按文件定位”，也服务“按关系阅读”。
- 备选：仅文件树。被拒绝原因是无法体现 OpenSpec 文件关系价值。

### Decision 4: 先采用轻量索引机制
- 方案：首版采用读取时扫描 + 显式重扫（CLI 或 Web 触发）机制；不引入复杂实时监听。
- 原因：快速交付且行为可预期，足以支撑只读预览。
- 备选：全量实时监听。被拒绝原因是复杂度高、跨平台行为差异大。

### Decision 5: UI 层级遵循“读 > 选 > 控”
- 方案：界面优先展示阅读对象（Capability / Change / File），其次是路径选择，最后才是系统控制操作（bind / unbind / stop service）。
- 原因：该产品本质是审阅工作台而非运维控制台，需要降低“操作面板感”，强化“阅读面板感”。
- 具体约束：
  - 高频对象区域（Change、Capability、File）应占首屏可见面积至少 70%。
  - 低频系统操作（unbind、stop service）进入次级入口（例如 workspace 菜单），不作为主按钮常驻主视觉区。
  - `Bind Workspace` 与 `Refresh Index` 组合为统一的 Sync Center，避免用户在不同位置重复理解同步入口。
  - 视觉权重递减：阅读对象 > 导航切换 > 系统控制；危险操作仅在触发前后显著提示。
  - 控件密度收敛：默认文本 13px，辅助文本 11-12px，常规按钮高度约 28px（主按钮 30px）。

### Decision 6: 引入可感知的同步状态与时间线
- 方案：以事件流方式记录 bind/refresh/change/capability 变化，并在界面提供“当前同步状态 + 最近历史”。
- 原因：Spec 推进不是静态阅读，用户需要知道“刚刚发生了什么、是否成功、影响了什么”。
- 具体约束：
  - Sync Center 展示 `idle/syncing/success/failed` 状态和最近一次摘要。
  - 时间线最小事件集包含：bind、refresh、change 更新、capability 更新。
  - 卡片需展示最近更新时间与关键事件摘要，提升可操作性与信息密度。

### Decision 7: 中文优先与品牌可替换
- 方案：默认使用中文界面文案；品牌区域支持优先渲染外部 logo，缺失时回退文字标题。
- 原因：目标用户以中文协作为主，且需要在不同团队环境中快速接入品牌资产。
- 备选：纯英文 + 固定文本品牌。被拒绝原因是本地协作语境与品牌接入成本不匹配。

### Decision 8: Capability 条目交互语义统一
- 方案：将导航区能力条目与右侧全局能力条目统一为同一“整行可点击 Row”样式，统一 hover/active/focus 状态与元信息布局。
- 原因：当前两处条目视觉不一致，用户难以判断“可点击对象”与“当前选中对象”，造成阅读动线中断。
- 具体约束：
  - 能力条目整行可点击，点击热区覆盖至少 90% 可见行宽。
  - 选中态在多区域同步高亮（同一 workspace 上下文）。
  - 时间元信息右对齐或末位对齐，避免标题换行破坏扫描效率。

### Decision 9: 操作分级与交互分区（Workspace Local vs Global Sync）
- 方案：将工作区局部操作与全局同步操作进行位置分区：
  - `刷新当前工作区` 放入工作区选择控件邻域（workspace local）。
  - `添加工作区 / 刷新全部 / 解绑全部` 归入同步中心（global sync）。
  - `解绑全部` 从工作区局部菜单迁移，避免局部区域承载全局危险操作。
- 原因：操作范围与按钮位置对齐后，用户更容易建立“在哪里做什么”的心智模型，减少误操作和视觉干扰。
- 具体约束：
  - Workspace 区域只承载当前上下文操作和局部菜单（如解绑当前）。
  - Sync Center 入口降权：采用紧凑 icon-first 样式，文案从“中心”语义降噪为工具语义。
  - 全局危险操作需保留二次确认，且在弹层中分组到 danger 区。
  - 高频操作按钮优先 icon 呈现，并通过 tooltip/辅助文案补足可理解性。

## UI 验收准则

- 5 秒可读性：首次进入页面 5 秒内，注意力应优先落在 Change / Capability / File，而非控制按钮。
- 路径效率：开始审阅的主路径不超过 2 步；低频系统控制路径不少于 2 步。
- 分区清晰：不读文字时也可明显区分“阅读工作区”与“系统控制区”。
- 空态体验：空数据场景依然保持阅读容器感，不退化为工具按钮集合。
- 一致性：Story/Compare 与 Navigation 模式切换时，阅读对象上下文保持连续，不发生无预期跳转。
- 状态可感知：同步动作均有可见状态反馈与历史记录，不出现“操作后无反馈”的盲区。
- 选择可见性：能力条目在不同区域的可点击性与选中态表达一致，用户无需学习两套交互语义。
- 操作分区清晰：用户在 3 秒内可区分“局部工作区动作”与“全局同步动作”，且无明显重复入口。

## Risks / Trade-offs

- [索引过期风险] 目录内容变化后页面可能短暂显示旧数据 → 提供显式“刷新索引”操作与最后扫描时间提示。
- [目录可达性风险] 已绑定目录可能被删除或权限变化 → 在目录列表保留绑定状态并标记错误，不自动丢失配置。
- [阅读复杂度风险] 并排与串读模式并存可能提高学习成本 → 提供默认 Story 视图，其他模式作为按需切换。
- [多工作区噪音风险] 工作区过多导致导航拥挤 → 支持按工作区过滤、最近访问优先展示。

## Migration Plan

1. 先交付只读版本：目录绑定、索引扫描、关系化阅读主流程。
2. 以真实使用反馈优化“待审阅项”的判定规则（保持规则可解释，不引入复杂分数）。
3. 在稳定后评估是否需要增量监听与更细粒度关联跳转。

## Open Questions

- “待审阅项”的最小判定集合是否固定为：新变更、未读变更、工件不完整三类。
- 并排对照默认组合是否固定为 `design vs tasks`，还是记忆用户最近选择。
